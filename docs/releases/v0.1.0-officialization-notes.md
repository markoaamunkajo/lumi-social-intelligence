# Lumi Social Intelligence 0.1.0 — Officialization Notes

This note complements the existing public technical release:

```text
tag: v0.1.0
status: existing public technical release
```

It does not replace the tag history and does not claim a newly published runtime.
It explains what has changed product-wise during the officialization sprints.

> We are making the self-improvement loop something that is human-shaped.

## Technical release exists

The public `v0.1.0` release already exists as a technical artifact for Lumi
Social Intelligence: source, adapters, release gates, public-safe artifact
builder, and dry-run preview materials.

That technical release should be understood as the public doorway, not proof of a
live autonomous Lumi service.

## Product-readiness delta

The officialization work adds a clearer product contract around the release:

| Sprint | Product-readiness contribution |
|---|---|
| Sprint 1 | Research harness contract for inspectable preview records |
| Sprint 2 | Ordered preview-loop protocol with consent checkpoints |
| Sprint 3 | Signal/reflection schema separating observation from interpretation |
| Sprint 4 | Consent, memory-boundary, and safety contract |
| Sprint 5 | Evaluation and acceptance criteria for “good enough” |
| Sprint 6 | Release-note packaging contract and publication review gate |

## What official 0.1.0 means here

Official 0.1.0 means:

- the preview loop is documented and testable;
- self-improvement proposals are inspectable;
- consent and memory boundaries are explicit;
- acceptance criteria are measurable;
- release notes distinguish technical artifacts from product-readiness evidence;
- publication/editing of public release notes remains a human decision.

## Safety boundaries

The officialization work remains review-only:

```text
canonical_writes: 0
runtime_actions: []
requires_human_review: true
```

It does not include:

- a live Lumi background runtime;
- Telegram bot/sender wiring;
- Hermes scheduler/config/memory mutation;
- credential handling;
- automatic durable memory promotion;
- extraction of private thinking-space content into company/product truth;
- autonomous public posting or GitHub release editing.

## Human review next step

Before any public release text is edited or republished, a human should review:

1. whether the officialization notes are accurate;
2. whether the existing GitHub release should be edited, supplemented, or left as-is;
3. whether the repo should tag a future product-readiness release separately;
4. whether any wording could imply hidden runtime capability that does not exist.
