# v0.2 Demo Index

This is the public-safe landing page for the current `v0.2` demo evidence path.

The demo is intentionally modest and receipt-first:

```text
synthetic/public-safe context → appraisal → presence decision → receipt
```

It exists to make the current proof easy to inspect without reading the whole build history.

## Current artifacts

| Artifact | Purpose |
|---|---|
| [`v0.2-demo-evidence.json`](v0.2-demo-evidence.json) | Machine-readable receipt for the underlying review-gated flow. |
| [`v0.2-demo-side-by-side.json`](v0.2-demo-side-by-side.json) | Machine-readable split between observed behavior and shadow-only behavior. |
| [`v0.2-demo-side-by-side.md`](v0.2-demo-side-by-side.md) | Human-readable Markdown view for demos, release notes, and copy-paste review. |

## What this demo currently shows

| Observed in this repository | Shadow-only / not yet claimed live |
|---|---|
| Review card generated from synthetic/public-safe context. | Native outbound emoji reaction delivery. |
| Context, appraisal, Presence decision, and receipt are connected. | Telegram send/reaction side effects. |
| Safety boundaries are recorded in machine-readable artifacts. | Durable memory promotion or canonical host writes. |

## Live-claim boundary

Native outbound reaction delivery is **not claimed live** by this repository yet.

Current public evidence keeps the boundary explicit:

```text
Native outbound reaction delivery: not_claimed
Canonical writes: 0
Telegram reactions sent: 0
```

- Canonical writes: `0`
- Telegram reactions sent: `0`
- The demo uses synthetic/public-safe context only.
- The repository does not read private chat logs, local scheduler internals, credentials, chat IDs, or raw runtime state.

## Regenerate the demo artifacts

```bash
python3 scripts/build_v02_demo_evidence.py \
  --report docs/demos/v0.2-demo-evidence.json \
  --side-by-side-report docs/demos/v0.2-demo-side-by-side.json \
  --side-by-side-markdown docs/demos/v0.2-demo-side-by-side.md
```

## Release rule

A `v0.2` demo can describe behavior as live only when the receipt contains observed evidence for that behavior. Until then, it must stay explicitly labeled as `shadow_only` or `not_claimed`.
