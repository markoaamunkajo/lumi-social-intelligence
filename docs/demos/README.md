# v0.2 Demo Evidence Path

This folder contains the public-safe evidence path for the `v0.2` demo track.

The goal is intentionally modest: prove a review-gated flow from selected context to Nuances-style appraisal to Presence-style output, while keeping live platform claims honest.

```text
Lumi Layered Memory -> Nuances -> Presence -> reviewable receipt
```

## Current demo receipt

The current receipt and side-by-side report are generated by:

```bash
python3 scripts/build_v02_demo_evidence.py \
  --report docs/demos/v0.2-demo-evidence.json \
  --side-by-side-report docs/demos/v0.2-demo-side-by-side.json
```

It records:

- the selected synthetic memory context;
- the Nuances appraisal for why the moment matters;
- the review-gated Presence decision;
- the reaction-aware Presence record;
- the outbound emoji Presence shadow record;
- the separation between observed behavior and shadow-only behavior.

## Current demo artifacts

- [`v0.2-demo-evidence.json`](v0.2-demo-evidence.json) is the machine-readable receipt for the underlying review-gated flow.
- [`v0.2-demo-side-by-side.json`](v0.2-demo-side-by-side.json) is the demo-facing report that separates observed behavior from shadow-only behavior.

## What is observed now

- A review card can be generated from public-safe synthetic context.
- The receipt can connect context, appraisal, Presence decision, and safety boundaries.
- Tests verify the contract fails closed when a live native reaction is implied without observed evidence.

## What remains shadow-only

- Native outbound emoji delivery in Telegram is not claimed as live behavior by this repository yet.
- The repository does not send Telegram messages or reactions.
- The repository does not read private runtime state.
- The repository does not promote observations into durable memory.

## v0.2 release rule

A `v0.2` demo can describe behavior as live only when the receipt contains observed evidence for that behavior. Until then, it must stay explicitly labeled as `shadow_only`.
